# M6：银行对账工具 — 开发踩坑记录

> 来源：实际开发过程中 5 轮评审修复的详细记录。每条坑都有"为什么会踩 → 怎么发现的 → 怎么修的"完整链条。

---

## 坑 1：CSV ArrayBuffer 直接当字符串用 → 崩溃

**现象**：用户上传 CSV 文件后，工具直接报错崩溃。

**原因**：`handleCSVUpload` 中把 `FileReader` 读到的 ArrayBuffer 当字符串调用了 `.includes()` 方法。ArrayBuffer 没有这个方法。

**教训**：文件上传不是"拿到文件对象就完了"，ArrayBuffer → 字符串需要显式解码。写文件处理函数时，先确认每一步的数据类型。

**修复**：用 `TextDecoder` 先将 ArrayBuffer 解码为 UTF-8 字符串，再做后续处理。

---

## 坑 2：ID 碰撞——银行和账务各自从 0 编号

**现象**：出现重复金额时（如两边各有一个 -23 元的快递费），工具无法正确标记候选匹配。

**原因**：银行记录和账务记录各自从 0 开始编号，导致 `id=3` 可能同时指向一条银行记录和一条账务记录。用 `bankById.has(id)` 判断时无法区分来源。

**教训**：跨数据源的 ID 必须全局唯一。最简单的方案：在 ID 上追加 `source` 字段（`bank-3` / `acct-3`）。

**修复**：每条记录新增 `source: 'bank' | 'accounting'` 字段，所有查询改用 `source + id` 组合键。

---

## 坑 3：列映射保存了但不恢复

**现象**：用户每次上传文件后都要重新选一遍列映射，明明上次已经选过了。

**原因**：`localStorage` 保存了映射关系（`persistMapping`），但 `updateColumnSelects` 函数没有读取已保存的映射来初始化下拉框。保存和恢复是两个独立函数，后者没调用前者保存的数据。

**教训**：持久化的"保存"和"恢复"必须成对验证。每写一个 `saveX()` 就要追问：对应的 `restoreX()` 在哪里？什么时候被调用？

**修复**：`updateColumnSelects` 优先读取 `localStorage` 中已保存的映射，并加上表头指纹校验（表头变了 → 忽略旧映射）。

---

## 坑 4：双列检测被"上次没检测到"的记忆抑制

**现象**：第一次上传的文件是单列金额，不显示"借贷分列"选项。第二次上传的文件是借贷分列格式（借/贷各一列），工具依然不显示双列选项。

**原因**：`localStorage` 保存了 `dualAmount: false`（上次检测结果），新文件加载时读到这个 `false`，就不激活双列检测逻辑了。保存的布尔值抑制了新的结构性检测。

**教训**：**结构性检测结果不能被保存状态抑制**。文件结构（单列/双列）是每次上传都重新检测的，不能因为"上次是单列"就跳过检测。只有用户偏好（如列映射选择）才应该持久化。

**修复**：双列检测始终在当前文件上执行，不受 `saved.dualAmount` 影响。

---

## 坑 5：列索引恢复不校验越界

**现象**：用户上一次选了"金额"列在第 7 列。换了一个只有 5 列的新文件后，工具尝试用 `columns[7]`，越界崩溃。

**原因**：`localStorage` 保存的列索引（如 `amountIndex: 6`）直接应用到新文件，没有检查新文件的实际列数。

**教训**：从持久化状态恢复的任何索引/序数值，必须先校验范围 `[0, currentCount)`。这就是"位置型状态越界"——保存时有效，恢复时上下文已变。

**修复**：恢复列索引前检查 `savedIndex < currentFileColumns.length`，越界则回退到自动检测。

---

## 坑 6：DOM 状态在同一会话中累积

**现象**：先用"粘贴模式"录入数据，切换到"上传模式"后，前一个模式留下的复选框状态还在（借贷分列 checkbox 仍选中）。

**原因**：`updateColumnSelects` 只在"确认需要"时才设置 UI 状态，但从未"重置不需要的"状态。模式切换时，旧 DOM 状态残留。

**教训**：同会话内的 UI 状态切换，**必须先重置所有条件性 DOM 状态为默认值，再根据当前数据激活需要的部分**。不能用"上次没这个功能所以没改"的逻辑。

**修复**：`updateColumnSelects` 函数开头统一重置 `dualCheckbox`、`dualRow`、`signFlip` 为默认值，然后根据当前文件再条件激活。

---

## 坑 7：拖拽上传区域闪烁

**现象**：拖拽文件到上传区时，高亮框反复闪烁，体验很差。

**原因**：拖拽事件（`dragenter`/`dragleave`）在子元素之间传递时会触发多次。直接用 `classList.toggle` 会在每次 enter 加、每次 leave 减，导致拖到上传区内部子元素时高亮消失。

**教训**：拖拽事件用计数器模式而非 toggle 模式。维护一个 `dragCounter`，只在 0→1 时高亮，1→0 时取消高亮。

**修复**：用 `dragCounter` 替代直接 toggle。

---

## 总结：7 个坑的通用模式

| 类型 | 坑 | 通用原则 |
|------|-----|---------|
| 数据类型 | #1 ArrayBuffer → 崩溃 | 文件处理每一步确认数据类型 |
| ID 设计 | #2 各自编号 → 碰撞 | 跨数据源 ID 必须全局唯一 |
| 持久化对账 | #3 保存 ≠ 恢复 | 每个 saveX() 必须有 restoreX() |
| 保存状态优先级 | #4 旧状态抑制新检测 | 结构性检测 > 保存状态 |
| 索引安全 | #5 列索引越界 | 恢复的索引必须校验范围 |
| 状态重置 | #6 DOM 残留 | 先全量重置，再条件激活 |
| 事件处理 | #7 拖拽闪烁 | 计数器模式替代 toggle 模式 |
